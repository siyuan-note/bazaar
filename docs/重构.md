# Bazaar重构

### 格式与存储

- [X] 根目录的集市包列表 JSON 改为 TXT，避免开发者 PR 时因尾随逗号产生冲突 [fd78924](https://github.com/siyuan-note/bazaar/commit/fd789242e704b0d0f97bfbaf74cf3415803698e9)

  形式如下：

  ```
  owner1/repo
  owner2/repo
  ```

  解析为：

  ```json
  [
    "owner1/repo",
    "owner2/repo"
  ]
  ```

  - [X] 工作流读取 TXT 后按行解析为 `[]string`​，校验每行为合法 `owner/repo`；校验失败时 Stage 报错退出、PR Check 回复一条错误消息
  - [X] README 详细说明 PR 方式步骤

  > [!NOTE]
  > 未来[支持将集市包标记为“弃用”](https://github.com/siyuan-note/siyuan/issues/16816)时就单独增加一个 deprecated.json 文件（解析为 map）：
  >
  > ```json
  > {
  >   "owner1/repo": {
  >     "xxx": "xxx"
  >   },
  >   "owner2/repo": {
  >     "xxx": "xxx"
  >   }
  > }
  > ```
  >
- [ ] Stage 工作流始终基于 package.zip 更新版本，不再依赖仓库文件和版本标签 [*](https://github.com/siyuan-note/bazaar/issues/1699)

  流程：

  - [X] 获取 Latest Release 的 tag 对应的 commit hash，与旧 hash 对比（如果没有持久化记录的话就单独存一个文件），如果相同则跳过（不需要下载 package.zip）
  - [ ] 下载 Latest Release 的 package.zip 并解压
  - [ ] 检查根目录（如果只有一个文件夹话，将这个文件夹视为根目录）是否存在必要的文件（不同类型的集市包有不同的必要的文件，弄几个不同的文件数组分类验证；比如 index.css 对插件来说不是必要的，因为思源有处理不存在 index.css 的情况）（Stage 和 PR Check 的文件检查不同，Stage 只要保证最基本的文件即可，PR Check 要更严格）（文件名大小写敏感）
  - [ ] 递归检查所有的目录和文件的名称是否规范（不能以空格开头或结尾、不能使用保留字 checkReservedWindowsDeviceName）
  - [ ] 获取与解析该类型集市包的元数据 JSON 文件
  - [ ] 收集元数据

    - [ ] 检查元数据中必要的字段是否完整、字段值符合要求（比如 `disabledInPublish`​ 字段严格来说可以不添加，但如果添加了就要检查是否是 bool 类型）（不同类型的集市包有不同的字段要求）（参考我目前的集市审核流程来写检查）

      - [ ] `name`

        - [X] 首次：验证字段是否合法，不能有前后空白字符……
        - [ ] 非首次：跟旧信息的字段值完全一致；PR Check 无效时回复具体原因：[https://github.com/siyuan-note/bazaar/pull/1115](https://github.com/siyuan-note/bazaar/pull/1115)
      - [ ] `url`​：等于 `https://github.com/owner/repo`​（避免斜杠结尾、`.git` 结尾）
      - [ ] `version`​：是否比旧信息的 `version`​ 字段版本更高（使用与思源内核一致的 "golang.org/x/mod/semver" 库判断[语义化版本](https://semver.org/lang/zh-CN/)，只要这个库能判断的就算）
      - [ ] `readme`：声明的 README 文件是否存在
    - [ ] 需要移除原来的 `sanitizePackage`​ 函数（用户下载到本地的 JSON 文件是原样的，只在 stage 做消毒没有用），改成对 `displayName`​ 和 `description`​ 字段值执行 `html.EscapeString`​（[#16975](https://github.com/siyuan-note/siyuan/pull/16975) 集市重构也配合修改）：

      ```go
      // sanitizePackageDisplayStrings 对集市包直接显示的信息做 HTML 转义，避免 XSS。
      func sanitizePackageDisplayStrings(pkg *Package) {
      	if pkg == nil {
      		return
      	}
      	for k, v := range pkg.DisplayName {
      		pkg.DisplayName[k] = html.EscapeString(v)
      	}
      	for k, v := range pkg.Description {
      		pkg.Description[k] = html.EscapeString(v)
      	}
      }
      ```
    - [ ] 精简 Stage JSON，减少数据传输流量消耗、提高集市列表加载速度

      - json 字段的排序要固定，避免只是字段排序变化就重新上传 stage json
      - `readme`：要填 default 值（如果有 en-us 没有 default 的话，就把 default 的值设置为 en-us 的值）、移除所有跟 default 值相同的其他字段；过滤掉包里没有对应文件的字段（default字段最后回退到README.md，如果这个也没有就只能删掉整个readme字段了，但要看看内核能不能处理没有 readme 字段的情况）
      - 重构为使用 map 而不是 struct，仅收集有非空值的字段、只添加必要的非空字段、过滤掉不在规范中的字段（具体哪些空值字段可以简化，还需要看旧版的内核实现，不能破坏掉旧版的逻辑。看看内核是否必须要有什么字段，还是序列化的时候会自动补上空值）（其他为 null 或空值的字段，在旧版内核里因为是结构体，或者判断了 nil，所以简化也完全没问题）

      比如：

      ```json
      "readme": {
        "default": "",
        "ar_SA": "",
        "de_DE": "",
        "en_US": "README.md",
        "es_ES": "",
        "zh_CN": "README.zh_CN.md"
      },
      "funding": {
        "openCollective": "",
        "patreon": "",
        "github": "",
        "custom": null
      }
      ```

      改成：

      ```json
      "readme": {
        "en_US": "README.md",
        "zh_CN": "README.zh_CN.md"
      }
      ```
  - [ ] 收集并上传文件

    - 上传 package.zip
    - 按照 `readme` 字段声明来收集 README 文件

  注意所有文件相关的逻辑都是大小写敏感的，因为思源内核没有在 Mac 和 Linux 上做大小写兼容。

  以上任何流程失败则立即跳过该仓库检查、回退到旧的 Stage 信息、不上传新的文件到 OSS，然后在 bazaar 仓库回复一个特定的 issue 输出该仓库检查中出现的所有错误（每个仓库给一条固定的回复），等到某次检查之后不出现错误了就可以自动删除该条回复。等功能稳定之后还可以添加自动 @ 仓库维护者进行处理的功能。

  > [!NOTE]
  > OSS 文件路径形式：
  >
  > package.zip 文件：`https://oss.b3logfile.com/package/TCOTC/Whisper@752988333ee14c0529d7b166d5ba5d378173c513`
  >
  > README.md 文件：`https://oss.b3logfile.com/package/TCOTC/Whisper@752988333ee14c0529d7b166d5ba5d378173c513/README.md`
  >

### 审核自动化

- [ ] PR Check 改成像 ob 那样，只把出错的地方列出来（比如把错误信息存在结构体里，按顺序格式化输出，不需要复杂的 Go 模板）

  - [ ] 工作流给 PR 创建两类消息，包含 PR 主要信息的消息固定在顶上，只有一个，每次检查完了更新；审核错误消息每次都输出一个新的在最后
- [ ] 根据我目前的集市审核流程来写检查（JSON 检查部分可以让 PR Check 和 Stage 工作流共用代码：逻辑可能不同，但代码要集中在一起）

  - [ ] PR Check 验证 PR 本身修改的 `owner/repo` 结构是否正确、是不是有效的 GitHub 仓库格式
  - [ ] 添加/更改维护者集市包的数量只能是一个（需要单独审核），不需要运行检查，直接回复需要单独创建多个 PR。移除集市包不限数量
  - [ ] 如果检查到是更换维护者，则回复相关说明的链接
  - [ ] bazaar 的变更如果只有一个仓库的话那就自动修改PR标题（包括更改维护者）

    比如 Add owner/repo，再添加类型标签 plugin

    如果添加了多个新仓库，要提示开发者拆分到不同的 PR
  - [ ] 验证集市包根目录是否有必要的文件（文件名大小写敏感）
  - [ ] PR Check 要验证集市包的 JSON 结构是否正确

    不能有多余的字段，否则以后官方要新增字段的话会跟现有的集市包冲突
  - [ ] 重名检查包含所有类型的 name
- [ ] PR Check 无论怎样都要输出回复，不能直接 exit 1 退出。网络错误之类的情况也要提示手动重试检查
- [ ] 对插件引入源代码静态分析（先把插件开发规范做出来，再根据文档里的规范来写规则）

  参考：[Add static code analysis for plugins in PR check by Copilot · Pull Request #9 · TCOTC/bazaar](https://github.com/TCOTC/bazaar/pull/9)

  这个功能必须是一个独立的组件，不要耦合到工作流中，因为对闭源插件的分析要在本地运行。
